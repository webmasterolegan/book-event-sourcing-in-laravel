# Саги (Sagas)

Вернемся к главе 10, где мы обсуждали концепцию реакторов для управления потоком программы. Я упоминал, что реакторы могут использоваться для создания так называемых саг или сложных процессов. Эта глава будет посвящена таким потокам.

Давайте начнем с рассмотрения части заказа на нашей диаграмме событий. Вы видите, что все, связанное с заказом, запускается автоматически через реакторы.

![Диаграмма, показывающая, что OrderCreated запускает CreateOrder (реактор), который, вероятно, создает что-то еще.](/images/18.png)

Однако на этой диаграмме все еще кое-чего не хватает: что должно произойти, когда заказ завершен? Мы должны сделать две вещи: сгенерировать PDF-счет и отправить его по электронной почте клиенту.

Поскольку мы не можем отправить письмо до того, как PDF будет сгенерирован, этот процесс следует рассматривать как "единое целое". Это поднимает интересный вопрос: что, если генерация PDF выполняется асинхронно? Может быть, даже есть выделенный сервис, независимый от нашей основной программы. Хотя "генерация PDF" и "отправка письма" являются единым процессом, оказывается, что сам этот процесс разделен на несколько более мелких частей.

Именно такого рода проблему и призвана решать сага.

Начнём с добавления `OrderCompletedReactor` — реактора, который будет контролировать весь процесс в целом.

![Сага/реактор OrderCompletedReactor, который слушает OrderCompleted, затем выдает команду CreateOrderInvoice, после чего слушает OrderInvoiceCreated, затем выдает команду SendOrderMail, после чего слушает OrderMailSent, и, возможно, логирует.](/images/19.png)

Что этот реактор должен сделать, так это выдать две команды: сначала создать PDF, затем отправить письмо. На нашей диаграмме это выглядело бы примерно так:

![](/images/20.png)

Возможно, вы уже видите, к чему мы идем: наш реактор будет слушать не только событие `OrderCompleted`, но и события `OrderInvoiceCreated` и `OrderMailSent`: снаружи эти события представляют один процесс, поэтому их можно обрабатывать в одном реакторе.

Наш реактор выглядел бы примерно так:

```php
class OrderCompletedReactor extends Reactor
{
    public function onOrderCompleted(OrderCompleted $event): void
    {
        $order = Order::find($event->orderUuid);
        (new CreateOrderInvoice)($order);
    }

    public function onInvoiceCreated(OrderInvoiceCreated $event): void
    {
        $order = Order::find($event->orderUuid);
        (new SendOrderMail)($order, $event->invoicePath);
    }

    public function onMailSent(OrderMailSent $event): void
    {
        Log::info("Письмо для заказа {$event->number} было отправлено.");
    }
}
```

Как видите, основная идея саг не нова. Мы просто немного иначе структурируем код. Поступая так, мы получаем несколько преимуществ.

**Во-первых**, саги могут управлять своим внутренним состоянием, так же как корни агрегатов. Вы можете прослушивать события, сохранять некоторые их данные в переменных класса и принимать решения, основанные на этом состоянии, позже в процессе.

**Во-вторых**, вы могли бы ввести способ сериализации саг и их сохранения на диск. Представьте процесс, который может занять несколько дней и который иногда требует принятия решения для продвижения вперед. Вы могли бы присвоить саге уникальный идентификатор и сохранить ее, чтобы позже иметь возможность ее извлечь.

**Наконец**, саги могли бы обрабатывать функциональность отката, если что-то пошло не так где-то в процессе. Если эти ошибки представлены в виде событий, вы могли бы подключиться к ним и, например, выдать несколько команд, чтобы сбросить состояние базы данных в то состояние, в котором оно было до начала процесса, или, возможно, уведомить нужного человека о сбое.

Помните, что есть способы решить наш первоначальный пример без использования саг: мы могли бы иметь один метод реактора, который прослушивает событие OrderCompleted, и который генерирует PDF и отправляет письмо последовательно. Этого решения определенно достаточно, если вы работаете в централизованной кодовой базе.

Однако саги имеют свое место в сложных приложениях, особенно если вы работаете в распределенной системе, где нет другого способа, кроме асинхронной обработки событий. Они являются хорошим решением для определенного круга проблем.
