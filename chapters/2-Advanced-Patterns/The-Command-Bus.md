# Шина команд (Command Bus)

До этого момента мы избегали хорошо известной парадигмы в событийно-ориентированных системах. Причина в том, что у нас просто не было в ней необходимости, в основном потому, что все главы до сих пор предполагали, что мы работаем в одном приложении с линейным программным потоком. В этой главе мы рассмотрим то, что мы упустили — как преимущества, которые это дает, так и добавленные трудности. Мы собираемся обсудить команды и обработчики команд.

В некотором смысле, мы уже использовали концепцию, похожую на команды и обработчики, с самого начала этой книги — мы называли их "действиями". И у команд, и у обработчиков те же цели:

- перенести бизнес-функциональность подальше от точек входа, таких как контроллеры или CLI-команды;
- визуализировать намерение нашего кода.

Действие по сути инкапсулирует все необходимое для того, чтобы в приложении что-то произошло. Извне — контроллеры или CLI-приложения — нам не интересно, как выполняется действие, мы просто хотим, чтобы оно произошло и вернуло правильный результат. Например, проще сказать "добавить товар в корзину", чем "извлечь корень агрегата корзины, добавить в него товар, сохранить его, а затем извлечь спроецированные результаты".

Команды служат той же цели, но отличаются от действий тем, что команда и ее обработчик разделены на отдельные классы. Так же, как и события, команды — это простые объекты данных, в то время как их обработчик знает, как с ними работать. Для сопоставления команды с ее обработчиком используется шина команд, точно так же, как шина событий.

Зачем нужна эта дополнительная сложность? Этот косвенный подход предлагает те же преимущества, что и события с шиной событий: он разделяет нашу систему на более слабо связанные компоненты, которые легче масштабировать. Конечно, ваш проект может не нуждаться в этой дополнительной гибкости, поэтому я ждал до сих пор, чтобы поднять эту тему. Вполне возможно, что вам никогда не понадобится шина команд, которая, например, может обрабатывать команды асинхронно или отправлять их на другие серверы, если вы создаете распределенную систему.

Прежде чем продолжить обсуждение преимуществ и трудностей шины команд, давайте посмотрим, как она выглядит на практике. Вот команда, которая добавляет товар в корзину:

```php
namespace Spatie\Shop\Cart\Commands;

use Spatie\Shop\Support\EventSourcing\Attributes\AggregateUuid;
use Spatie\Shop\Support\EventSourcing\Attributes\HandledBy;
use Spatie\Shop\Support\EventSourcing\Commands\Command;

#[HandledBy(CartAggregateRoot::class)]
class AddCartItem implements Command
{
    public function __construct(
        #[AggregateUuid]
        public string $cartUuid,
        public string $cartItemUuid,
        public Product $product,
        public int $amount,
    ) {}
}
```

Обратите внимание, что вы можете проверить ветку command-bus демо-приложения, чтобы увидеть ее реализацию для агрегата корзины.

Вы могли заметить, что команды могут автоматически сопоставляться с корнем агрегата с помощью атрибутов. Это, по сути, делает метод агрегата обработчиком команды. Эта реализация была вдохновлена Axon, популярным фреймворком event sourcing в мире Java. Это ярлык для того, чтобы не писать вручную множество обработчиков, которые всегда делают одно и то же: извлекают агрегат, вызывают метод и сохраняют изменения.

За кулисами эта команда будет сопоставлена с CartAggregateRoot, используя `$cartUuid` в качестве идентификатора. Она будет искать метод в корне агрегата, который обрабатывает эту команду, на основе ее типа.

Здесь мы подошли к важному преимуществу использования шины команд. Команды также работают с частями агрегатов, а это означает, что нам больше не нужно ручное сопоставление между методами корня агрегата и методами части агрегата. Помните, как мы предоставляли "прокси-методы" в корне агрегата в предыдущей главе? Они выглядели так:

```php
class CartAggregateRoot extends AggregateRoot
{
    protected CartItems $cartItems;

    //...

    public function addItem(
        string $cartItemUuid,
        Product $product,
        int $amount
    ): self {
        // $this->cartItems->addItem(
        //     $cartItemUuid,
        //     $product,
        //     $amount,
        // );

        return $this;
    }

    public function removeItem(CartItem $cartItem): self
    {
        // $this->cartItems->removeItem($cartItem);
        return $this;
    }
}
```

Шина команд, предоставляемая пакетом shop, достаточно умна, чтобы знать, что команды также должны передаваться в части агрегатов, а это означает, что нам больше не нужно предоставлять эти прокси-методы. Наша часть агрегата CartItems будет выглядеть так:

```php
class CartItems extends AggregatePartial
{
    // ...

    public function addItem(AddCartItem $addCartItem): self
    {
        // ...
    }

    public function removeItem(RemoveCartItem $removeCartItem): self
    {
        // ...
    }
}
```

И CartAggregateRoot больше не нуждался бы в этих методах `addItem()` и `removeItem()`.

Всякий раз, когда команда отправляется, шина команд автоматически определит правильный метод-обработчик на основе ее атрибутов и подсказки типа в корне агрегата (или одном из его частей агрегата). Таким образом, отправка команды становится такой же тривиальной, как:

```php
class CartController
{
    public function addCartItem(Request $request, CommandBus $commandBus): void
    {
        $commandBus->dispatch(new AddCartItem(
            cartUuid: /*...*/,
            cartItemUuid: /*...*/,
            product: /*...*/,
            amount: /*...*/
        ));
    }
}
```

Теперь, когда основы рассмотрены, есть еще несколько вещей, которые нужно обсудить.

Во-первых, всегда существует отображение "один к одному" между командами и их обработчиками. Это потому, что обработчику разрешено возвращать результат, указывающий, была ли команда успешно обработана или нет. Однако работа с корнями агрегатов event sourcing в качестве обработчиков команд делает это немного сложнее, поскольку сами корни агрегатов не должны ничего возвращать.

Например, мы не сможем напрямую вернуть результат, такой как проекция; но мы можем генерировать исключения, чтобы указать, что что-то пошло не так. Вы уже знаете о косвенном характере событийно-ориентированного дизайна, поэтому вы знаете, что есть как преимущества, так и недостатки.

На практике это означает, что нам придется разрешать ожидаемый результат после выполнения команды вручную:

```php
class CartController
{
    public function pay(Request $request, CommandBus $commandBus): void
    {
        $paymentUuid = PaymentUuid::make();

        $commandBus->dispatch(new PayCart(
            cartUuid: /*...*/,
            paymentUuid: $paymentUuid,
        ));

        $payment = Payment::find($paymentUuid);
    }
}
```

Кроме того, имейте в виду, что побочные эффекты команд могут быть поставлены в очередь (например, проекторы или реакторы). Это означает, что мы не сможем немедленно вернуть результат, как мы делали в примере выше. Например, представьте, что создание платежа включает связь с внешним поставщиком платежей. Это может занять несколько секунд или даже минут.

Решение таких проблем требует от нас создания интерфейса с асинхронным мышлением. Наивным решением может быть ожидание в контроллере перед возвратом:

```php
class CartController
{
    public function pay(Request $request, CommandBus $commandBus): void
    {
        $paymentUuid = PaymentUuid::make();

        $commandBus->dispatch(new PayCart(/*...*/));

        while (! $payment = Payment::find($paymentUuid)) {
            continue;
        }

        return $payment;
    }
}
```

Хотя можно утверждать, что это довольно сомнительное решение: что, если внешний поставщик платежей в данный момент недоступен? Будем ли мы просто ждать ошибки тайм-аута? Это не очень удобное для пользователя решение. Лучшее решение в этом случае будет зависеть от бизнес-требований, но, скорее всего, будет включать одно из двух:

- опрос на стороне клиента в сочетании с удобным для пользователя экраном загрузки, пока платеж обрабатывается;
- или использование формы асинхронной связи, такой как электронная почта, чтобы сообщить клиенту, что его платеж готов и может быть оплачен.

Вы видите, что не существует "технически правильного" ответа на эти вопросы, так как все будет зависеть от бизнес-требований. Однако мы знаем, что наш контроллер не должен ждать возврата результата. Вместо этого мы вернем UUID платежа, чтобы фронтенд мог решить, что с ним делать:

```php
class CartController
{
    public function pay(Request $request, CommandBus $commandBus): void
    {
        $paymentUuid = PaymentUuid::make();

        $commandBus->dispatch(new PayCart(/*...*/));

        return $paymentUuid;
    }
}
```

Еще одна деталь, заслуживающая внимания: у команд всегда должен быть обработчик. Если команда остается необработанной, мы генерируем исключение. Это мера безопасности для программистов, поскольку косвенный поток таких систем может затруднить обнаружение того, действительно ли команда обрабатывается или нет.

Далее, шины команд также поддерживают middleware. Middleware позволяет нам внедряться в поток шины команд и добавлять всевозможную функциональность. Например, вы могли бы регистрировать каждую отправленную команду. Такое middleware выглядело бы примерно так:

```php
use Spatie\EventSourcing\Commands\Middleware;

class LogMiddleware implements Middleware
{
    public function handle(object $command, Closure $next): mixed
    {
        $commandClass = $command::class;
        Log::info("Команда {$commandClass} была отправлена.");
        return $next($command);
    }
}
```

Добавление middleware в шину команд должно быть выполнено при ее регистрации в контейнере зависимостей:

```php
class AppServiceProvider extends ServiceProvider
{
    // ...

    public function register(): void
    {
        $this->app->singleton(CommandBus::class, function () {
            $commandBus = new CommandBus();
            $commandBus->middleware(new LogMiddleware());
            return $commandBus;
        });
    }
}
```

Другое, более сложное middleware предоставляется пакетом event sourcing: RetryMiddleware. Оно повторяет любую неудачную команду заданное количество раз перед ее окончательным сбоем. Это особенно полезно, если существует высокая вероятность одновременных запросов к одному и тому же экземпляру корня агрегата.

Представьте, что наша корзина может быть изменена несколькими людьми одновременно. Мы должны убедиться, что операции с нашим корнем агрегата корзины выполняются синхронно, в противном случае наш хронологический журнал событий может быть поврежден. Итак, что произойдет, если два человека захотят внести изменения в этот корень агрегата одновременно?

Что ж, одна из команд завершится ошибкой, потому что пакет защищает от такого рода состояний гонки. Но что теперь? Вероятно, вы захотите повторить неудачную команду, поскольку есть большая вероятность, что она завершится успешно, даже с изменениями, внесенными в агрегат предыдущей командой. Итак, мы хотим повторно извлечь корень агрегата (чтобы иметь самую последнюю версию) и отправить неудачную команду еще раз.

Это то, что RetryMiddleware делает за кулисами. Это очень простой подход к обработке конфликтов параллелизма, но во многих случаях также эффективный.

## ТЕСТИРОВАНИЕ

Как и события, сами команды не должны тестироваться, если они содержат только данные. Обработчики, конечно, должны, и наличие шины команд позволяет упростить некоторые тесты.

Представьте сложный процесс, состоящий из нескольких команд. Вы бы тестировали каждый обработчик команды индивидуально, но вы также хотели бы знать, все ли команды запускаются в правильном порядке. С шиной команд у вас есть одно центральное место, куда прибывают все команды, и, следовательно, место, где вы можете утверждать, какие команды были запущены, а какие нет.

Вы могли бы, например, создать тестовую шину команд, которая не только отправляет команды, но и ведет журнал всех команд, которые прошли через нее, в порядке. Вы могли бы прочитать этот журнал (это может быть массив имен команд) и делать утверждения на его основе.

```php
public function test_complex_process
{
    $this->commandBus->dispatch(new StartProcess(/*...*/));

    $this->assertEquals(
        [
            StartProcess::class,
            SubCommandA::class,
            SubCommandB::class,
            SubCommandC::class,
            EndProcess::class,
        ],
        $this->commandBus->getLog()
    );
}
```

Возможно, что шины команд являются излишними для вашего проекта. Если вы работаете над монолитным проектом, то от них мало пользы. Кроме того, имейте в виду, что у использования команд есть своя цена: появляется больше классов и больше косвенности. Для некоторых проектов эта цена оправдана, но не для всех.
