# Шина событий (Event Bus)

Прежде чем узнать обо всем, что возможно с событийно-ориентированной системой, нам нужно понять две самые основные концепции, лежащие в ее основе. Начнем с шины событий.

Думайте о шине событий как о позвоночнике любой событийно-ориентированной системы. Это ядро, которое знает обо всех слушателях и о том, какие события они обрабатывают.

В своей простейшей форме шина событий — это класс всего с несколькими строками кода: он отслеживает, какие слушатели зарегистрированы для каких событий, и уведомляет их всякий раз, когда событие отправлено. Это действительно не более чем:

```php
class EventBus
{
    private array $listeners = [];

    public function addListener(string $eventName, callable $listener): self
    {
        $this->listeners[$eventName][] = $listener;
        return $this;
    }

    public function dispatch(object $event): void
    {
        $listenersForEvent = $this->listeners[$event::class] ?? [];

        foreach ($listenersForEvent as $listener) {
            $listener($event);
        }
    }
}
```

Эта реализация слишком проста, чтобы быть готовой к продакшену. Настоящая шина событий, вероятно, должна:

- уметь обрабатывать события асинхронно;
- поддерживать middleware, чтобы разработчики могли вклиниться в процесс отправки событий;
- вероятно, знать, как работать с другими типами слушателей, кроме простых callable-объектов и т.д.

Хотя правильная шина событий сама по себе является интересным инфраструктурным программным обеспечением, она не будет в центре внимания этой книги. Каждый пакет или фреймворк event sourcing уже предоставит шину событий, которая обладает большинством описанных выше функций. В этой книге мы больше заинтересованы в том, что мы можем сделать с шиной событий.

Итак, начиная с этого момента, мы будем считать, что шина событий дана нам и нам не нужно больше о ней беспокоиться. Если вы хотите немного погрузиться в код, чтобы узнать больше, я рекомендую начать с пакета League/event. Это правильная реализация шины событий, из которой можно многому научиться.

## ТЕСТИРОВАНИЕ

Поскольку мы предполагаем, что шина событий всегда является частью фреймворка, мы не будем обсуждать тестирование ее внутреннего устройства.

На протяжении всей этой книги мы увидим множество примеров событий и слушателей, и проверка того, правильно ли они связаны, является важным шагом в разработке нашего приложения.

Тестирование слушателей изолированно возможно путем создания мока шины событий и наблюдения за тем, какие слушатели были вызваны, но следует подумать, добавляют ли такие тесты много ценности. Позже в этой книге мы узнаем, как балансировать между модульным и интеграционным тестированием, и последние уже будут проверять нашу конфигурацию слушателей.

Нам не следует повторять тесты для одного и того же, так что давайте пока воздержимся от создания моков шины событий и посмотрим, куда нас приведут интеграционные тесты в следующих главах.
