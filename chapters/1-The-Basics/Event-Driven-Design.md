# Событийно-ориентированный дизайн

Однажды у меня был спор с одним из моих коллег о событийно-ориентированных системах. Я сказал ему, что мне не нравится работать с событиями, потому что это неявный способ программирования. Он понял мою точку зрения, но все же сказал, что стоит в этом разобраться.

Моя проблема с событийно-ориентированными системами заключалась в том, что всякий раз, когда вы отправляете событие, вы не можете сразу увидеть, что произойдет дальше. Возьмем этот пример:

```php
dispatch(new CartCheckedOut($cart->uuid));
```

Поскольку одно событие может обрабатываться несколькими слушателями, вам пришлось бы найти всех потребителей этого события, чтобы понять, что именно происходит при отправке CartCheckedOut. Возможно, есть слушатель, который отправляет письмо клиенту, возможно, генерируется PDF-счет, или, возможно, изменяется инвентарь добавленных в корзину товаров. Невозможно сказать, что именно произойдет, не зная и не проверяя всех слушателей такого события.

Это неудобный способ программирования — или так я думал.

По иронии судьбы, именно косвенный характер событийно-ориентированных систем открывает мир возможностей. В программировании это часто так и есть. Чем больше гибкости требуется, тем сложнее становится решение.

Давайте начнем с детального изучения этой одной строки кода: "оформление заказа". Мы начнем с обычной реализации и будем продвигаться к событийно-ориентированной, обсуждая плюсы и минусы. Допустим, "оформление заказа" имеет два побочных эффекта:

1.  Обновляется инвентарь товаров.
2.  PDF-счет отправляется клиенту.

Один из способов спроектировать процесс оформления заказа — создать отдельные функции или классы для каждого шага этого процесса и объединить их в классе, предназначенном для "оформления заказа". В таких случаях мне нравится начинать изнутри наружу.

Например, мы могли бы создать один класс, отвечающий за обновление инвентаря:

```php
class UpdateInventory
{
    public function __invoke(Cart $cart): void
    {
        foreach ($cart->items as $item) {
            $item->product->inventory->subtract($item->amount);
        }
    }
}
```

И другой класс, отвечающий за отправку счета клиенту:

```php
class SendInvoice
{
    public function __construct(
        private GeneratePdf $generatePdf,
    ) {}

    public function __invoke(Cart $cart): void
    {
        $pdf = ($this->generatePdf)(
            new InvoiceTemplate($cart)
        );

        $cart->customer->notify(
            new InvoiceForCustomer($cart, $pdf)
        );
    }
}
```

Наконец, мы можем объединить их в классе, который отвечает за "оформление заказа":

```php
class CheckoutCart
{
    public function __construct(
        private UpdateInventory $updateInventory,
        private SendInvoice $sendInvoice,
    ) {}

    public function __invoke(Cart $cart): void
    {
        $cart->update([
            'status' => CartStatus::CHECKED_OUT,
            'checked_out_at' => now(),
        ]);

        ($this->updateInventory)($cart);
        ($this->sendInvoice)($cart);
    }
}
```

Я предпочитаю разделять эту функциональность на отдельные классы и внедрять их с помощью автопроводки Laravel, потому что в реальности такой пример, как этот, имеет тенденцию быть больше и сложнее. Если мы сохраним весь код в одном классе, он будет расти и расти, что затруднит его тестирование, а также повысит вероятность ошибок, если нам потребуется что-то в нем изменить.

Я описывал эту концепцию ранее в книге "Laravel Beyond CRUD" и называл их actions (действиями): классами, которые выполняют определенные бизнес-операции.

Однако с действиями есть проблема: как и при размещении всего кода в одном классе, действия, составленные из других действий, также могут стать довольно большими, но без чьего-либо ведома. Если мы не будем осторожны, существует вероятность получить глубокую цепочку зависимостей, которую очень трудно поддерживать. Даже в нашем текущем примере мы уже приближаемся к пределу. Давайте визуализируем дерево зависимостей:

Я не показывал код действия GeneratePdf, но вы можете представить, как он использует внешнюю библиотеку для генерации PDF из, например, HTML-шаблона.

Даже с этим упрощенным примером мы уже имеем дело с четырехуровневой цепочкой зависимостей. А теперь позвольте мне показать вам реальный пример: это (слегка упрощенное) дерево зависимостей класса в проекте, над которым я работал в прошлом:

![Сложное дерево зависимостей глубиной 7 уровней. Текст: "Цель всего этого кода — сохранить счет в базе данных, асинхронно создать платеж через внешнего провайдера и отправить пользовательское письмо со ссылкой на оплату через Mandrill."](/images/4.png)

Цель всего этого кода — сохранить счет в базе данных, асинхронно создать платеж через внешнего провайдера и отправить пользовательское письмо со ссылкой на оплату через Mandrill.

Вы никогда не стали бы писать код, вложенный на семь уровней вглубь, но поскольку в этом примере классы внедряются, их вложенные зависимости скрыты из виду, если вы не начнете их искать. Кроме того, эти действия очень жестко связаны между собой. Даже несмотря на то, что функциональность разделена между классами, снаружи это все равно один монолитный, тесно связанный кусок кода.

Увидев такой граф зависимостей, я задумался об ограничениях того, что я считал "более простым подходом". Внезапно косвенность событийно-ориентированных систем открыла множество возможностей. Если нам не нужно беспокоиться о микроменеджменте следующих шагов в потоке программы, мы получим гораздо больше гибкости, а также будем писать более устойчивый код.

Итак, давайте переосмыслим дизайн: что, если "оформление заказа" — это событие, в которое может подключиться столько частей, сколько мы захотим? Вместо того чтобы микроменеджерить все шаги, связанные с нашим процессом выставления счетов, мы можем зарегистрировать наши "действия" как обработчики событий.

Возможно, вы удивитесь, но внесенные нами изменения минимальны. Давайте посмотрим на новый UpdateInventory, который теперь живет в пространстве имен App\Listeners:

```php
namespace App\Listeners;

// ...

class UpdateInventory
{
    public function handle(CartCheckedOut $event): void
    {
        foreach ($event->cart->items as $item) {
            $item->product->inventory->subtract($item->amount);
        }
    }
}
```

И это действие SendInvoice, теперь живущее в пространстве имен App\Listeners:

```php
namespace App\Listeners;

// ...

class SendInvoice
{
    public function __construct(
        private GeneratePdf $generatePdf,
    ) {}

    public function handle(CartCheckedOut $event): void
    {
        $pdf = ($this->generatePdf)(
            new InvoiceTemplate($event->cart)
        );

        $cart->customer->notify(
            new InvoiceForCustomer($event->cart, $pdf)
        );
    }
}
```

Может показаться, что мы только добавили кода, но имейте в виду, что наше действие CheckoutCart теперь может быть написано так:

```php
class CheckoutCart
{
    public function __invoke(Cart $cart): void
    {
        $cart->update([
            'status' => CartStatus::CHECKED_OUT,
            'checked_out_at' => now(),
        ]);

        dispatch(new CartCheckedOut($cart));
    }
}
```

Но кроме перемещения кода, что мы выиграли? Что ж, давайте еще раз посмотрим на нашу схему:

![Диаграмма, показывающая, как диспетчеризация события развязывает связи. Действие CheckoutCart отправляет событие, а слушатели (UpdateInventory, SendInvoice) подписываются на него.](/images/5.png)

Нам не нужно беспокоиться о конкретных побочных эффектах в любой части нашего кода, и это дает нам огромную свободу. То, что мы обсудили в этой главе, является основой для многих будущих принципов. Событийно-ориентированная архитектура даст нам свободу и гибкость. Она сделает сложные проблемы тривиальными для решения, позволит нам естественным образом масштабировать наш проект и многое другое. Все это построено на простом принципе событий.
