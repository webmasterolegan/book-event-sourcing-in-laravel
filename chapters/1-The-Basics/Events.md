# События (Events)

В отличие от шины событий, сами события будут в центре нашего внимания на протяжении всей этой книги. Крайне важно понять, что такое событие, поэтому мы посвятим этому целую главу.

По сути, событие — это не более чем утверждение о том, что "в системе произошло что-то". Это сообщение, которое рассылается для всех, кто заинтересован его услышать. Это означает, что "событие" по определению не обязательно привязано только к нашей собственной программе: сложным системам может потребоваться транслировать события во внешний мир, потому что другие системы их слушают.

Но мы забегаем вперед; сначала нам нужно договориться о деталях, составляющих событие. В некоторых случаях самого события достаточно, чтобы слушатели могли его обработать. Другими словами: событие могло бы быть "пустым сообщением". Тот факт, что сообщение отправлено, достаточен. Мартин Фаулер дает таким событиям подходящее название Event Notification (Уведомление о событии): события, которые используются только как сигнал для своих слушателей.

Такое событие было бы таким же простым, как:

```php
class CartCheckedOut {}
```

Вы сразу заметите, что трудно придумать хороший вариант использования таких событий: событие CartCheckedOut без указания того, какая корзина была оформлена, бесполезно. Однако вы могли бы спроектировать системы, которые слушали бы эти уведомления и, в свою очередь, искали бы в базе данных или где-то еще, что именно изменилось.

Тем не менее, у вас, вероятно, такие же сомнения, как и у меня: события без какой-либо контекстной информации не кажутся такими уж полезными.

Вот почему большинство событий несут в себе некоторое состояние. Если события — это сообщения, то их данные представляют собой тело сообщения. Мартин Фаулер называет этот принцип Event-Carried State Transfer (Передача состояния через событие): событие содержит некоторую информацию, чтобы сделать его более полезным. В случае нашего события CartCheckedOut мы могли бы добавить ID корзины, чтобы указать, о какой корзине идет речь:

```php
class CartCheckedOut
{
    public function __construct(
        public int $cartId,
    ) {}
}
```

Обратите внимание, что я предпочитаю использовать публичные свойства в событиях. В наших проектах уже давно существует соглашение о том, что данные событий доступны только для чтения, и в нашей команде мы согласны соблюдать это правило без программного принуждения. Однако если вы не чувствуете уверенности в таком дизайне, вы можете добавить геттеры и сделать свойства событий закрытыми.

Передача состояния через событие поднимает вопрос: какую информацию мы отправляем с событиями и, что более важно, какую не отправляем? Не могли бы мы отправить всю модель корзины вместо только ее ID?

Давайте посмотрим на пример, который прояснит проблему такого подхода. Представьте событие InvoiceGenerated, которое срабатывает, когда генератор PDF-счетов завершил свою работу. Отправили бы вы сам PDF-файл через событие или скорее его путь? Если мы используем события только как внутрипроцессный механизм для отправки сообщений в нашей собственной кодовой базе, то ответ был бы "скорее всего, и то, и другое". Но что, если бы мы отправляли наши сообщения в другие сервисы или, возможно, сохраняли сериализованные версии событий в базе данных? Тогда мы не смогли бы отправить сам PDF через событие, так как это привело бы к значительным проблемам с памятью.

Такого рода проблемы могут показаться пограничным случаем, но есть один способ использования событий, о котором я еще не упомянул, и где вопрос о том, как сериализовать события, становится очень важным: event sourcing.

Под event sourcing мы подразумеваем, что сообщения в нашей системе не просто транслируются и забываются. Вместо этого каждое событие сохраняется в какой-либо базе данных или где-то еще на диске. Первое преимущество этого — мы получаем журнал всех событий, произошедших в системе. Но что более важно, event sourcing использует события не как механизм для обработки вторичной функциональности, как мы часто видим в Laravel; вместо этого сами события становятся центральным механизмом нашей системы.

Взгляните на наше предыдущее действие CheckoutCart:

```php
class CheckoutCart
{
    public function __invoke(Cart $cart): void
    {
        $cart->update([
            'status' => CartStatus::CHECKED_OUT,
            'checked_out_at' => now(),
        ]);

        dispatch(new CartCheckedOut($cart));
    }
}
```

Здесь мы отправляем событие CartCheckedOut только для обработки побочных эффектов оформления заказа (обновление инвентаря и отправка PDF-счета клиенту). Что, если мы перевернем ситуацию, и "обновление статуса корзины" также будет считаться частью этих "побочных эффектов"?

Тогда наше действие CheckoutCart выглядело бы так:

```php
class CheckoutCart
{
    public function __invoke(Cart $cart): void
    {
        dispatch(new CartCheckedOut(
            cartId: $cart->id,
            checkoutDate: now(),
        ));
    }
}
```

А обработчик, предназначенный для обновления корзины, выглядел бы так:

```php
class CheckoutCartProjector
{
    public function handle(CartCheckedOut $event): void
    {
        $cart = Cart::find($event->cartId);
        $cart->update([
            'status' => CartStatus::CHECKED_OUT,
            'checked_out_at' => $event->checkoutDate,
        ]);
    }
}
```

Я понимаю, что здесь происходит несколько вещей, которые могут показаться странными. Что это за CheckoutCartProjector? Почему дата оформления заказа вдруг передается в событие? Мы скоро ответим на эти вопросы и другие. Сначала важно понять вторую ключевую часть event sourcing. Если события хранятся в базе данных, и если все в нашем приложении является побочным эффектом этих событий, что произойдет, если мы удалим все наши данные, кроме событий, и воспроизведем все события заново? Отправим их снова, чтобы запустить все побочные эффекты еще раз.

Представьте, что также есть события CartCreated и CartItemAdded, сохраненные в базе данных со всеми соответствующими данными. Мы могли бы удалить нашу корзину и восстановить ее на основе информации, сохраненной в событиях. Такая коллекция сохраненных событий называется потоком событий (event stream): это хронологический список событий, которые принадлежат друг другу, например, события, составляющие одну корзину.

Сравните этот подход с традиционными CRUD-приложениями. Самое важное отличие заключается в том, что каждое CRUD-действие перезаписывает существующие данные. Event sourcing только добавляет новые данные, записывая новые события поверх старых. Вы никогда не теряете информацию о том, что произошло в прошлом. Если вы хотите достичь того же результата в CRUD-приложениях, вам нужно явно предоставить инфраструктуру логирования или истории, чтобы отслеживать, что произошло, в то время как при event sourcing это дано с самого начала.

Может показаться, что от наличия такого встроенного журнала мало пользы, но на протяжении всей этой книги мы откроем для себя несколько мощных паттернов, основанных на этом простом принципе. Я уверен, что по мере продвижения вы начнете ценить мощь event sourcing.

Есть несколько замечаний. При event sourcing приходится иметь дело со множеством тонкостей: неужели мы не хотим отправлять PDF-счет клиенту второй раз при перестроении приложения? Как отслеживать инкрементные идентификаторы, если все удаляется? Как сделать события сериализуемыми, чтобы их можно было хранить в базе данных?

Предстоит охватить много материала, когда дело доходит до event sourcing, и нам понадобится большая часть этой книги, чтобы со всем этим справиться, но уже важно понимать преимущества, которые он дает:

- Сохраненные события — это журнал всего, что произошло в вашей системе.
- События можно воспроизводить, то есть вы можете восстанавливать старое состояние или строить новое состояние на основе старых событий постфактум.
- Моделирование с помощью событий — это естественный способ "мышления о мире" — мы исследуем эту тему в следующей главе.

## ТЕСТИРОВАНИЕ

Пока мы соблюдаем строгие границы, определяющие событие — простой класс данных без какой-либо логики, нет причин писать для них явные тесты. То есть: только если вы используете строго типизированные свойства.

Я большой сторонник полного использования системы типов PHP, а также всего, что доступно в мире статического анализа, чтобы сократить количество необходимых тестов. Глубокое погружение в тему статического анализа выходит за рамки этой книги, но если вы хотите прочитать больше об этом, вы можете прочитать мою книгу "Front Line PHP".
