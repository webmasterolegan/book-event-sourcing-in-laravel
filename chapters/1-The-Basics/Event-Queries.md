# Событийные запросы (Event Queries)

Хотя проекции являются мощным инструментом, у них есть и недостатки. Представьте, что ваш клиент просит разные виды отчетов. В этом случае может быть некоторая нагрузка на создание специальной проекции для каждого отдельного отчета. Если к тому же отчеты должны быть фильтруемыми, вы можете получить множество итераций одной и той же проекции.

Возьмем такой пример: ваш клиент хочет получить отчет о доходах от продуктов за определенный период. Другими словами, и продукт, и период — это две переменные. Было бы невозможно генерировать строку для каждой возможной комбинации периода и продукта.

Один из способов решения такой проблемы с помощью проекций — иметь нормализованный набор данных и группировать их на лету. Вы могли бы хранить проекцию, которая перечисляет доходы по продуктам за день, и фильтровать и агрегировать эти данные на основе запроса пользователя.

Это правильный подход к проблеме, хотя есть более простое решение, которое обходится без использования промежуточной таблицы проекций: событийные запросы (event queries).

Событийный запрос — это класс, который собирает группу событий и воспроизводит их на лету. Разница с воспроизведением проекторов в том, что событийный запрос отслеживает "состояние приложения" исключительно в памяти: он ничего не хранит в базе данных. Каждый раз, когда вы выполняете событийный запрос, он заново перестраивает свое внутреннее состояние. Этот принцип позволяет вам создавать высоконастраиваемый запрос, который может предоставлять результаты, основанные на потребностях пользователя.

Давайте начнем с нашего класса событийного запроса:

```php
namespace App\Reports;

use Illuminate\Support\Collection;
use Spatie\EventSourcing\EventHandlers\Projectors\EventQuery;
use Spatie\EventSourcing\StoredEvents\Models\EloquentStoredEvent;
use Spatie\Period\Period;

class EarningsForProductAndPeriod extends EventQuery
{
    public function __construct(
        private Period $period,
        private Collection $products,
    ) {
        // ...
    }
}
```

Всякий раз, когда мы создаем новый событийный запрос, мы хотим загрузить связанные события в память. Мы уже видели, что события хранятся в модели EloquentStoredEvent, и что мы можем использовать ее для запроса наших сохраненных событий:

```php
public function __construct(
    private Period $period,
    private Collection $products,
) {
    EloquentStoredEvent::query()
        ->whereEvent(OrderCreated::class)
        ->whereDate('created_at', '>=', $this->period->getStart())
        ->whereDate('created_at', '<=', $this->period->getEnd())
        ->cursor()
        ->each(fn (EloquentStoredEvent $event) => $this->apply($event->toStoredEvent()));
}
```

Поскольку в нашей базе данных могут храниться миллионы событий, лучше помнить о производительности:

- мы фильтруем события по их типу
- нас интересуют только события OrderCreated
- затем мы фильтруем по периоду, чтобы еще больше сузить наш запрос
- и, наконец, мы используем курсор для загрузки сохраненных событий постепенно, вместо того чтобы загружать их все в память сразу.

Когда все релевантные события доступны, мы проходим по ним и применяем их с помощью `$this->apply()`. Под капотом этот метод ищет обработчики событий в самом классе запроса и вызывает их так, будто событие только что было отправлено — во многом как при воспроизведении событий для проектора. Такой обработчик — это защищённый метод самого класса запроса, и выглядит он так:

```php
class EarningsForProductAndPeriod extends EventQuery
{
    // ...

    private int $totalPrice = 0;

    protected function applyOrderCreated(
        OrderCreated $orderCreated
    ): void {
        $orderLineData = collect(
            $orderCreated->orderData->orderLineData
        );

        $totalPriceForOrder = $orderLineData
            ->filter(function (OrderLineData $orderLineData) {
                return $this->products->first(
                    fn (Product $product) =>
                        $orderLineData->productEquals($product)
                ) !== null;
            })
            ->sum(
                fn (OrderLineData $orderLineData) =>
                    $orderLineData->totalPriceIncludingVat
            );

        $this->totalPrice += $totalPriceForOrder;
    }
}
```

Давайте пройдемся по этому шагу за шагом. Первое, что вы должны заметить: всякий раз, когда корзина оформляется, создается заказ. Вот почему мы слушаем событие OrderCreated: это доказательство того, что продажа действительно произошла.

Если события имеют тенденцию сильно разрастаться по данным, возможно, будет хорошей идеей использовать в них объекты передачи данных (DTO). Это именно то, что происходит с объектами OrderData и OrderLineData. Это простые объекты, содержащие данные, которые могут быть сериализованы.

Затем мы фильтруем нашу коллекцию строк заказа по продуктам в них. Нас интересуют только продукты, соответствующие нашему предопределенному фильтру.

Наконец, мы складываем все доходы от отдельных строк заказа вместе и увеличиваем наше внутреннее значение totalPrice. Помните, что мы применяем событие за событием, поэтому нам нужно отслеживать общую сумму, используя переменную экземпляра.

После применения всех событий остается только открыть внутреннюю totalPrice для внешнего мира:

```php
class EarningsForProductAndPeriod extends EventQuery
{
    // ...

    public function totalPrice(): int
    {
        return $this->totalPrice;
    }
}
```

Имея все это на месте, мы теперь можем генерировать эти отчеты на лету, в памяти:

```php
$report = new EarningsForProductAndPeriod(
    Period::make('2021-01-01', '2021-02-01'),
    $collectionOfProducts,
);

$report->totalPrice();
```

В некотором смысле вы можете думать о событийных запросах как о комбинации проекторов и проекций: вместо ручного воспроизведения событий в базу данных мы воспроизводим их на лету в памяти.

Событийные запросы — это не решение всех проблем, но они довольно полезны для решения конкретных вопросов, требующих относительно небольшого количества событий и большой гибкости. Также обратите внимание, что вы можете прослушивать несколько событий в одном событийном запросе и предоставлять любое состояние, которое захотите.

Событийные запросы являются изолированными компонентами, что упрощает их тестирование. Все, что нам нужно сделать, это подготовить несколько событий и утвердить, что правильный результат вычислен:

```php
/** @test */
public function total_price_is_calculated(): void
{
    $eventFactory = OrderCreatedFactory::new();

    $eventFactory->withTotalPrice(10_00)->onDate('2021-01-11')->create();
    $eventFactory->withTotalPrice(15_00)->onDate('2021-01-10')->create();
    $eventFactory->withTotalPrice(15_00)->onDate('2021-02-10')->create(); // Не должно учитываться

    $collectionOfProducts = collect([...]);

    $report = new EarningsForProductAndPeriod(
        Period::make('2021-01-01', '2021-02-01'),
        $collectionOfProducts,
    );

    $this->assertEquals(25_00, $report->totalPrice());
}
```

Мы продолжим развивать идею событийных запросов в будущей главе, но сначала мы рассмотрим критически важный недостающий элемент для управления потоком событий.
