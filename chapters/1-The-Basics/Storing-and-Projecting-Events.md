# Хранение и проектирование событий

Когда мы обсуждали шину событий в главе 3, я описывал, как она может поддерживать такие функции, как асинхронные события, middleware и т.д. Я не упомянул, что шина событий также является точкой входа для хранения и воспроизведения событий в приложении с event sourcing. В этой главе мы сделаем первые шаги к работе с сохраненными событиями.

Отныне наши примеры кода будут написаны в приложении Laravel с использованием пакета, предназначенного для event sourcing: spatie/laravel-event-sourcing. Этот пакет не нужен вам, чтобы следить за книгой; он лишь избавляет от скучного шаблонного кода, который в противном случае нам пришлось бы писать самим, например, шины событий, описанной выше. Этот пакет позволяет нам сосредоточиться на той части, где мы можем проектировать приложение на основе событий, вместо того чтобы иметь дело со всевозможным инфраструктурным кодом.

Мы закончили предыдущую главу диаграммой, которая выглядела так:

![Диаграмма со всеми событиями: CartInitialized, CartItemAdded, CartItemRemoved, CartCheckedOut, OrderCreated, PaymentCreated, PaymentSuccessful, PaymentFailed, OrderCompleted, OrderCancelled.](/images/11.png)

Прежде чем перевести эту диаграмму в код, лучше обсудить, как мы будем группировать код. Все на нашей диаграмме связано с корзинами, но на основе этих событий будет добавлено гораздо больше кода. Неплохо придумать какую-то категоризацию, чтобы отдельные части оставались поддерживаемыми. Мне нравится черпать вдохновение в сообществе DDD и делать так называемые агрегаты (aggregates): агрегат — это группа кода, которую можно рассматривать как "единое целое".

Отличным источником по теме агрегатов является книга Эрика Эванса "Предметно-ориентированное проектирование", а также Мартин Фаулер, который описывает их так:

>Агрегат DDD — это кластер объектов предметной области, которые можно рассматривать как единое целое. Примером может служить заказ и его позиции; это будут отдельные объекты, но полезно рассматривать заказ (вместе с его позициями) как единый агрегат.

Другими словами: агрегат представляет собой группу концепций, которые принадлежат друг другу, например, заказы и их позиции. Не имеет смысла работать с позициями заказа вне контекста заказов, поэтому заказ и его позиции можно рассматривать как единый агрегат.

Имея это в виду, мы можем определить четыре агрегата на нашей диаграмме:

- Корзина (cart), отвечающая за добавление и удаление товаров клиентами в корзину и из нее, вплоть до момента оформления заказа;
- Заказ (order), отвечающий за процесс, происходящий после оформления корзины;
- Инвентарь товара (product's inventory), который является второстепенной задачей, когда клиенты добавляют товары в корзину;
- Платеж (payment), который заслуживает собственного процесса, учитывая все изменения состояния, которые обычно с ним связаны.

Отражая эти изменения на диаграмме, мы получаем нечто подобное:

![Диаграмма с группировкой событий в агрегаты: "Корзина" (CartInitialized, CartItemAdded, CartItemRemoved, CartCheckedOut), "Платеж" (PaymentCreated, PaymentSuccessful, PaymentFailed), "Заказ" (OrderCreated, OrderCompleted, OrderCancelled), "Инвентарь" (возможно, пока без событий).](/images/12.png)

Пока мы используем агрегаты только как способ группировки кода, но в следующих главах мы сделаем еще несколько шагов. Мы присвоим этим агрегатам идентичность, чтобы все события были связаны с конкретным экземпляром агрегата. Представьте, что вы присваиваете ID модели Cart Eloquent. Вместо того чтобы этот ID идентифицировал только одну строку базы данных, он будет использоваться как идентификатор группы, агрегата в целом.

Аналогично, когда бы ни было отправлено событие, оно будет исходить из одного из этих агрегатов. Будет крайне важно иметь возможность проследить событие до его источника. Например: "это событие CartItemAdded связано с корзиной номер 1". Другими словами, это событие произошло от агрегата Cart, и конкретно того, у которого ID 1.

Говоря об этих ID, есть важное замечание. Мы установили, что события будут источником истины в нашем приложении, а это означает, что мы никогда не должны полагаться на данные, которые неизвестны событию. ID являются идеальным примером: если мы собираемся инициализировать корзину, ей обычно присваивается ID базой данных с использованием автоинкрементного ключа. Однако, если мы удалим наши проекции, чтобы восстановить их из потока событий, мы потеряем исходный ID, и база данных сгенерирует новый. Представьте, что были другие ресурсы, ссылающиеся на эту корзину по ее ID, который внезапно изменился, потому что мы решили перестроить все корзины. Это привело бы к многочисленным проблемам и ошибкам.

Помните наше одно ключевое правило, когда дело доходит до event sourcing: события — единственный источник истины. Если данных нет в событии, им нельзя доверять. На практике это означает, что мы будем использовать UUID вместо ID, потому что их гораздо проще генерировать извне — вы скоро увидите это на практике.

Далее, как мы видели в главе 4, мы должны тщательно продумывать, какие данные отправлять с событиями. Помните, мы собираемся сохранять эти сообщения в базе данных, поэтому крайне важно, чтобы они могли быть правильно сериализованы. Лучше не хранить сложные объекты целиком, а преобразовывать их в легко сохраняемый формат.

С учетом всего сказанного, давайте посмотрим, как выглядело бы наше первое событие, CartInitialized:

```php
<?php

namespace App\Cart\Events;

use Carbon\Carbon;
use Spatie\EventSourcing\StoredEvents\ShouldBeStored;

class CartInitialized extends ShouldBeStored
{
    public function __construct(
        public string $cartUuid,
        public string $customerUuid,
        public Carbon $date,
    ) {}
}
```

Обратите внимание на класс ShouldBeStored, от которого мы наследуемся, предоставленный пакетом event sourcing. Он добавляет функциональность для сериализации и десериализации сохраненных событий. Остальное довольно просто: это класс, содержащий некоторые данные, относящиеся к "моменту инициализации корзины". Одним из его свойств является UUID корзины; как мы видели ранее, важно хранить его в событии, чтобы иметь возможность перестроить состояние нашего приложения при необходимости.

Давайте посмотрим на другое событие, CartItemAdded:

```php
namespace App\Cart\Events;

use Spatie\EventSourcing\StoredEvents\ShouldBeStored;
use App\Product\Price;

class CartItemAdded extends ShouldBeStored
{
    public function __construct(
        public string $cartUuid,
        public string $cartItemUuid,
        public string $productUuid,
        public int $amount,
        public Price $currentPrice,
    ) {}
}
```

Помимо отслеживания UUID корзины и собственного UUID, обратите внимание, что мы также сохраняем текущую цену, а не только ссылку на сам товар. Представьте, что вы добавляете в корзину товар по сниженной цене, которую администратор меняет сразу после того, как вы его добавили. Вы все равно хотите, чтобы при оформлении заказа использовалась старая цена. По крайней мере, это бизнес-правило, которое я установил для этого примера. В реальной жизни могут быть еще более сложные бизнес-правила, поэтому сессии event storming так важны. Обсуждая поток приложения с вашим клиентом, такие правила обсуждаются и становятся ясными до того, как вы начнете писать код.

Далее, нам нужен способ отправки и хранения наших событий. Помните наши классы действий? Вот где нам нужно будет внести некоторые изменения. Например, вот действие для инициализации корзины:

```php
class InitializeCart
{
    public function __invoke(Customer $customer): Cart
    {
        $cartUuid = uuid();

        $event = new CartInitialized(
            cartUuid: $cartUuid,
            customerUuid: $customer->uuid(),
            date: now(),
        );

        $event->handle();

        return Cart::find($cartUuid);
    }
}
```

Здесь происходит несколько интересных вещей, давайте рассмотрим их шаг за шагом.

**Отправка события**

Мы вызываем `$event->handle()`, метод, предоставляемый пакетом в классе StoredEvent. Это не лучший способ отправки и хранения событий, и мы используем его только временно. В следующих главах мы откроем для себя лучшие способы управления потоком событий и соответствующим образом реорганизуем этот пример.

**Возврат корзины**

Где бы мы ни использовали это действие `InitializeCart`, скорее всего, мы захотим что-то сделать с корзиной после ее инициализации. Представьте, что это действие используется в контроллере, и мы хотели бы отправить UUID корзины и статус обратно клиенту.

Помните, что Cart — это проекция, побочный эффект события (мы подробно рассмотрим ее через минуту). Являясь проекцией, это означает, что мы можем немедленно вернуть Cart из этого действия только в том случае, если знаем, что события обрабатываются немедленно при отправке. Как мы увидим позже в этой книге, событийно-ориентированный дизайн подходит для создания асинхронных систем или распределенных систем, которые взаимодействуют друг с другом асинхронно. В этих случаях мы не сможем гарантировать, что проекция существует сразу после отправки события.

А пока мы будем считать, что работаем в синхронной системе и что все события обрабатываются немедленно. Это позволяет нам возвращать Cart непосредственно из нашего действия.

Что мы еще не обсуждали, так это то, как на самом деле создается такая проекция после отправки события.

Проекция — это таблица базы данных (или любой другой способ хранения данных), которая представляет собой один из способов просмотра потока событий. Это не обязательно должен быть единственный способ; это всего лишь одна из интерпретаций произошедших событий. В этом случае мы назовем проекцию Cart. Поскольку мы будем хранить проекции в базе данных, они будут обычными моделями Eloquent, как вы их знаете в типичном приложении Laravel. Обычными, но с одним отличием: проекции основаны на потоке событий, и нам не разрешено вносить в них изменения извне этого потока. Помните: события — единственный источник истины! Другими словами: каждое изменение наших проекций должно быть побочным эффектом событий.

Чтобы мы не забыли об этом правиле, пакет event sourcing представляет новый класс Projection. Этот класс наследуется от моделей Eloquent, как мы их знаем, с тем исключением, что он предотвращает прямое сохранение данных в них — я покажу, как они используются через минуту.

Давайте начнем с того, как выглядела бы сама проекция корзины:

```php
namespace App\Cart\Projections;

use Illuminate\Database\Eloquent\Relations\HasMany;
use Illuminate\Database\Eloquent\Relations\HasOne;
use Spatie\EventSourcing\Projections\Projection;
use App\Cart\States\CartState;

/**
 * @property string $uuid
 * @property int $total_price_excluding_vat
 * @property int $total_price_including_vat
 * @property CartState $state
 * @property-read \App\Shop\Customer\Customer $customer
 * @property-read \App\Shop\Cart\Projections\CartItem[] $items
 */
class Cart extends Projection
{
    protected $casts = [
        'total_price_excluding_vat' => 'integer',
        'total_price_including_vat' => 'integer',
        'state' => CartState::class,
    ];

    public function customer(): BelongsTo
    {
        return $this->belongsTo(Customer::class);
    }

    public function items(): HasMany
    {
        return $this->hasMany(CartItem::class);
    }
}
```

Как видите, обычная модель Eloquent. Обратите внимание, что я также добавил doc-блоки свойств для удобства. Если вы не уверены в классе `CartState`, вы можете прочитать о нем в документации [spatie/laravel-model-states](https://github.com/spatie/laravel-model-states) или в Laravel Beyond CRUD; короче говоря, это класс, который инкапсулирует всю логику, связанную с состоянием Cart.

Миграция нашей проекции выглядит так:

```php
Schema::create('carts', function (Blueprint $table) {
    $table->uuid('uuid');
    $table->uuid('customer_uuid');
    $table->foreign('customer_uuid')
          ->references('uuid')
          ->on('customers');
    $table->string('state');
    $table->integer('total_price_excluding_vat')->default(0);
    $table->integer('total_price_including_vat')->default(0);
    $table->timestamps();
});
```

Теперь, когда у нас есть место для хранения данных, нам понадобится место, куда можно подключиться к событиям и фактически сохранить проекцию. Классы, отвечающие за прослушивание событий и проецирование данных в базу данных, называются проекторами (projectors):

```php
namespace App\Cart\Projectors;

use Spatie\EventSourcing\EventHandlers\Projectors\Projector;
use App\Cart\Events\CartInitialized;
use App\Cart\Projections\Cart;
use App\Cart\States\PendingCartState;

class CartProjector extends Projector
{
    public function onCartInitialized(CartInitialized $event): void
    {
        (new Cart)->writeable()->create([
            'uuid' => $event->cartUuid,
            'state' => PendingCartState::class,
            'customer_uuid' => $event->customerUuid,
            'created_at' => $event->date,
        ]);
    }
}
```

В зависимости от того, как настроен пакет event sourcing, он может автоматически обнаруживать проекторы и подключать их в качестве слушателей к событиям. В этом случае вы можете предположить, что CartProjector автоматически регистрируется как класс-слушатель, и onCartInitialized будет вызываться всякий раз, когда отправляется событие CartInitialized.

Давайте отметим несколько вещей.

**Проекции, доступные для записи**

Как я уже говорил, проекции по умолчанию недоступны для записи в качестве механизма безопасности. Единственное место, где вы должны вносить изменения в проекции, — это проекторы. Вот почему вам нужно явно вызвать `writeable()` для корзины; в противном случае при попытке сохранить ее вы получите ошибку.

**Хранение данных**

Обратите также внимание, что мы не генерируем никаких новых данных, таких как ID или UUID, в проекциях: событие является источником истины и содержит UUID для этой корзины. То же самое относится и к `created_at`: мы не генерируем временную метку при создании проекции. Корзина была создана в момент отправки события, а не тогда, когда обрабатывался один из его побочных эффектов.

Представьте, если бы мы удалили эту проекцию Cart и перестроили ее через две недели после фактической инициализации корзины. Мы все равно хотели бы, чтобы ее дата `created_at` была той, когда событие было изначально отправлено, а не когда проекция была перестроена.

**Перестроение проекций**

Я уже несколько раз упоминал "перестроение проекций", но еще не объяснил, как это работает. Мы обсудим это в следующей главе.

Прежде чем закончить эту главу, давайте рассмотрим еще одну проекцию: добавление товара в корзину. Каждый раз, когда добавляется товар в корзину, мы должны добавить строку в таблицу `cart_items` и связать ее с корзиной, но мы также должны обновить общую стоимость корзины. Мы могли бы вычислять такого рода информацию на лету, когда работаем с корзиной, но я бы сказал, что лучше хранить ее в базе данных. Наличие такой информации позволяет нам напрямую запрашивать ее в базе данных и оказывает очень мало влияния на наш код.

По сути, можно сказать, что при добавлении товара в корзину есть два побочных эффекта: сам товар корзины должен быть добавлен, и цена корзины должна быть обновлена.

Обновления Cart должны обрабатываться нашим CartProjector:

```php
class CartProjector extends Projector
{
    // ...

    public function onCartItemAdded(CartItemAdded $event): void
    {
        $cart = Cart::find($event->cartUuid);

        $totalPriceExcludingVat = $cart->total_price_excluding_vat + $event->getTotalPriceExcludingVat();
        $totalPriceIncludingVat = $cart->total_price_including_vat + $event->getTotalPriceIncludingVat();

        $cart->writeable()->update([
            'total_price_excluding_vat' => $totalPriceExcludingVat,
            'total_price_including_vat' => $totalPriceIncludingVat,
        ]);
    }
}
```

Здесь мы видим проблеск силы косвенного характера событийно-ориентированных архитектур. Нам не нужен один большой класс, который знает обо всех шагах добавления товара в корзину. Вместо этого мы получаем несколько небольших и изолированных слушателей, которые обрабатывают свою конкретную часть потока программы.

Чем больше вы работаете с event sourcing, тем больше вы будете ценить такое поведение. Это чрезвычайно гибкий способ моделирования сложных потоков, сохраняя при этом отдельные части головоломки маленькими и краткими.

## ТЕСТИРОВАНИЕ

Я уже упоминал, что вы должны стремиться хранить в событиях только простые значения. Не только для упрощения их сериализации, но и для того, чтобы избежать необходимости тестировать различные геттеры или статические конструкторы.

Конечно, бывают случаи, когда такие методы действительно помогают, но помните, что их должно быть минимум. Если вы добавляете какую-либо функциональность в ваши классы событий, убедитесь, что вы написали для нее тест. Обычно это всего несколько строк кода:

```php
class CartItemAdded extends ShouldBeStored
{
    // ...

    public function getProduct(): Product
    {
        return Product::find($this->productUuid);
    }
}

/** @test */
public function test_get_product(): void
{
    $product = ProductFactory::new()->create();

    $event = new CartItemAdded(
        /* ... */,
        productUuid: $product->uuid,
    );

    $this->assertTrue($product->is($event->getProduct()));
}
```

Что касается тестирования проекторов, мы сначала рассмотрим их более подробно, а затем посмотрим на их тестирование.

Если вы следуете за книгой, не стесняйтесь реализовать удаление товаров из корзины самостоятельно. На этом мы закончим эту главу и продолжим более продвинутыми техниками проекторов в следующей.
