# Проекторы (Projectors) в деталях

До этого момента мы рассмотрели технические основы event sourcing: как хранить события и проецировать их. Возможно, у вас осталось чувство, действительно ли мы что-то выиграли, используя событийно-ориентированный подход. На первый взгляд может показаться, что мы добавили несколько дополнительных шагов для достижения той же цели.

Это восприятие изменится с этой главой: мы посмотрим, что позволяет делать такая простая концепция, как проекторы.

Сталкивались ли вы когда-нибудь с тем, что клиент спрашивает о новой функции, а вы должны были сказать ему, что она будет работать только с данными, добавленными после развертывания функции? Пример: представьте, что ваш клиент хочет провести углубленный анализ поведения клиентов. Возможно, они хотели бы выявить клиентов, которые оформили заказ менее чем за 10 минут. Что, если мы еще не отслеживали дату и время оформления заказа? Вам нужно было бы добавить новое поле даты оформления в корзину, которое будет равно null для всех корзин, существовавших до этой функции.

Однако при event sourcing у нас больше гибкости. Тот факт, что "событие произошло", — это информация, которая обычно отсутствует в традиционном приложении, где вы не отслеживаете каждое изменение. Система с event sourcing, с другой стороны, отслеживает, когда события были изначально отправлены. Это дает нам хронологический журнал всего, что произошло в нашей системе.

Таким образом, без необходимости изменения схемы базы данных мы можем решить проблему нашего клиента, потому что event sourcing также фиксирует суть времени. По моему опыту, именно эти чувствительные ко времени вопросы часто приносят наибольшую пользу бизнесу. Они позволяют нам отслеживать и сообщать о потоке нашего приложения во времени и помогают принимать решения на основе этих показателей постфактум.

Чтобы решить нашу проблему с клиентами, которые быстро оформляют заказ, есть два решения: мы могли бы добавить поле в проекцию Cart или создать специальную проекцию для такого рода отчета. Давайте начнем с первого.

Наш код работает в продакшене, и нам нужно добавить новое поле в существующую таблицу. Мы начнем с миграции:

```php
public function up()
{
    Schema::table('carts', function (Blueprint $table) {
        $table->dateTime('checked_out_at')->nullable();
    });
}
```

Далее, мы должны убедиться, что это поле также проецируется. Давайте перейдем к нашему CartProjector. Метод onCartCheckedOut уже был реализован:

```php
class CartProjector extends Projector
{
    // ...

    public function onCartCheckedOut(CartCheckedOut $event): void
    {
        $cart = Cart::find($event->cartUuid);
        $cart->writeable()->update([
            'state' => CheckedOutCartState::class,
        ]);
    }
}
```

Теперь нам нужно узнать, когда произошло событие. Наше событие само по себе не хранит эти данные (поскольку мы не добавили для них поле), но все же есть способ определить дату его создания. Когда бы событие ни было сериализовано в базу данных, оно оборачивается в модель EloquentStoredEvent — обычную модель Eloquent. Она не только хранит сериализованное событие, но и отслеживает некоторые метаданные, включая дату его создания. Если мы можем получить сохраненную модель события для этого события, мы можем получить доступ к ее данным. К счастью, каждый класс события автоматически получает некоторые метаданные, включая ID, который используется для его хранения в базе данных.

Зная все это, получение сохраненного события — не что иное, как обычный запрос к базе данных:

```php
$storedEvent = EloquentStoredEvent::find($event->storedEventId());
```

Собирая все вместе, мы реорганизуем наш проектор следующим образом:

```php
class CartProjector extends Projector
{
    // ...

    public function onCartCheckedOut(CartCheckedOut $event): void
    {
        $cart = Cart::find($event->cartUuid);
        $storedEvent = EloquentStoredEvent::find($event->storedEventId());

        $cart->writeable()->update([
            'state' => CheckedOutCartState::class,
            'checked_out_at' => $storedEvent->created_at,
        ]);
    }
}
```

Однако, мы еще не закончили! Если мы отправим эти изменения в продакшен, они повлияют только на корзины, которые были оформлены после нашего развертывания.

Помните, как я объяснял, что event sourcing позволяет нам перестраивать состояние нашего приложения с нуля? Мы могли бы, по сути, удалить все наши данные, кроме самих событий, и перестроить все заново. Это именно то, что мы собираемся сделать: удалить все корзины и воспроизвести события. Мы могли бы, в теории, удалить и перестроить все, но вы можете представить, сколько времени это займет. Поэтому вместо того, чтобы начинать с нуля, мы воспроизведем только события, связанные с корзинами.

Перед воспроизведением нам понадобится способ сбросить нашу таблицу корзин. У проекторов есть встроенное решение для этого: метод `resetState()`:

```php
class CartProjector extends Projector
{
    public function resetState(): void
    {
        Cart::query()->delete();
    }

    // ...
}
```

Этот метод выполняется всякий раз, когда мы воспроизводим проектор. Последнее, что нужно сделать после развертывания наших изменений, — это фактически воспроизвести этот проектор. Это делается с помощью следующей команды artisan:

```bash
php artisan event-sourcing:replay "App\\Cart\\Projectors\\CartProjector"
```

После завершения этой команды события будут воспроизведены, и проектор корзины запишет дату оформления заказа во все корзины, включая те, что были в прошлом.

Удаление всех корзин и их перестроение может стать довольно дорогостоящим с течением времени. Представьте себе производственную систему с годами данных; перестроение всех корзин может занять несколько часов, а то и дней. Вы не захотите, чтобы ваша система была в "режиме обслуживания" все это время.

Мы рассмотрим стратегии развертывания для таких случаев в следующей главе, но также стоит упомянуть другое решение.

Вместо изменения самой корзины мы могли бы создать новую проекцию, предназначенную для отчета, который запрашивает наш клиент. Они хотят знать, какие клиенты оформили свою корзину менее чем за 10 минут, поэтому давайте создадим специальную таблицу, которая отслеживает время активности корзины. Вот миграция:

```php
Schema::create('cart_durations', function (Blueprint $table) {
    $table->uuid('uuid');
    $table->uuid('cart_uuid');
    $table->dateTime('created_at');
    $table->dateTime('checked_out_at')->nullable();
    $table->integer('duration_in_minutes')->nullable();
});
```

И вот проекция:

```php
class CartDuration extends Projection
{
    protected $guarded = [];
    public $timestamps = [];

    protected $casts = [
        'created_at' => 'datetime',
        'checked_out_at' => 'datetime',
        'duration_in_minutes' => 'integer',
    ];
}
```

Наконец, нам нужно сделать проектор. Мы создадим новый класс, но будем прослушивать существующие события CartInitialized и CartCheckedOut:

```php
namespace App\Cart\Projectors;

// ...

class CartDurationProjector extends Projector
{
    public function onCartInitialized(CartInitialized $event): void
    {
        $createdAt = EloquentStoredEvent::find($event->storedEventId())->created_at;

        (new CartDuration)->writeable()->create([
            'uuid' => $event->cartUuid,
            'cart_uuid' => $event->cartUuid,
            'created_at' => $createdAt,
        ]);
    }
```

```php
    public function onCartCheckedOut(CartCheckedOut $event): void
    {
        $checkedOutAt = Carbon::make(
            $event->storedEventId()
        )->created_at;

        $cartDuration = CartDuration::find($event->cartUuid);
        $durationInMinutes = $checkedOutAt->diffInMinutes($cartDuration->created_at);

        $cartDuration->writeable()->update([
            'checked_out_at' => $checkedOutAt,
            'duration_in_minutes' => $durationInMinutes,
        ]);
    }
}
```

Всякий раз, когда корзина инициализируется, этот проектор создаст новую проекцию CartDuration и сохранит дату срабатывания события. Обратите внимание, что мы используем UUID корзины как UUID продолжительности. Поскольку Cart и CartDuration находятся в отношении один-к-одному, можно использовать его повторно. Технически вам даже не понадобился бы специальный UUID для этой проекции, но Laravel ожидает его наличия.

Затем, когда срабатывает событие CartCheckedOut, мы обновляем существующую CartDuration для этой корзины и вычисляем продолжительность в минутах. Когда этот проектор запускается для ранее отправленных событий, он создаст новую CartDuration для всех существующих корзин из-за события CartInitialized, а также вычислит продолжительность, когда эти корзины были окончательно оформлены.

Осталось только развернуть эти изменения и воспроизвести события на нашем новом проекторе:

```bash
php artisan event-sourcing:replay "App\\Cart\\Projectors\\CartDurationProjector"
```

Вы видите, как мы создали еще одну интерпретацию того же потока событий, создав новый проектор, который прослушивает те же события. Это еще раз иллюстрация того, как события являются единственным источником истины, а проекции — лишь побочный эффект.

Помимо возможности реагировать на события прошлого, есть еще одно преимущество, которое мы получили: после того как наши проекции `CartDuration` сгенерированы, их очень легко читать. Нам не нужно выполнять вычисления в памяти или соединения с базой данных: данные находятся прямо здесь, когда они нам нужны, в правильном формате. Это также означает, что вы могли бы создавать специальные проекции для конкретных страниц, даже если эти проекции основаны на одном и том же потоке событий. Нет необходимости динамически создавать панель администратора на основе спроецированных корзин. Давайте просто создадим специальную проекцию для этой панели.

Наконец, больше нет страха что-то упустить: вы можете добавлять любое количество проекций после того, как ваше приложение ушло в продакшен. Пока события хранятся, вы можете воспроизводить их столько раз, сколько захотите, и любым способом. Мы рассмотрим силу проекций даже глубже, когда будем обсуждать CQRS, тему для следующей главы.

## ТЕСТИРОВАНИЕ

Проекторы обычно не делают ничего большего, чем перемещение данных из событий в проекции. Классический тест проектора будет состоять из ручной отправки события проектору и утверждения того, что была создана правильная проекция.

```php
/** @test */
public function cart_initialized(): void
{
    $projector = new CartDurationProjector();
    $event = CartInitializedFactory::new()->create();

    $projector->onCartInitialized($event);

    $cartDuration = CartDuration::find($event->cartUuid);

    $this->assertNotNull($cartDuration);
}
```

В случае расчета продолжительности корзины есть немного больше для тестирования:

```php
/** @test */
public function cart_checked_out(): void
{
    $projector = new CartDurationProjector();

    $cartDuration = CartDurationFactory::new()
        ->createdAt(Carbon::make('2021-01-01 10:00'))
        ->create();

    $event = CartCheckedOutFactory::new()
        ->withCart($cartDuration->cart)
        ->createdAt(Carbon::make('2021-01-01 10:25'))
        ->create();

    $projector->onCartCheckedOut($event);

    $cartDuration->refresh();

    $this->assertEquals(25, $cartDuration->duration_in_minutes);
}
```

Что мы еще не протестировали, так это то, действительно ли эти проекторы слушают правильные события. К этому мы вернемся позже, когда будем писать интеграционные тесты.

## Подводные камни

Прежде чем закончить эту главу, я должен упомянуть несколько предостережений.

Первое, о чем я уже упоминал: воспроизведение событий может быть трудоемким. Бывают случаи, когда это не проблема, но иногда нужно знать, что ваша стратегия развертывания может стать более сложной.

Далее, есть еще вопрос о том, какие данные хранить в событиях. Хотя вносить изменения в проекции легко, вносить изменения в события (например, добавлять или удалять свойство) очень сложно. Изменение события может иметь серьезные последствия при их воспроизведении. Это также тема, которой мы посвятим главу позже в этой книге.

Несмотря на эти предостережения, я думаю, что использование системы с event sourcing имеет большую ценность. Хотя некоторые проблемы сложнее решить из-за event sourcing, другие становятся тривиальными. Мы посвятим будущие главы дальнейшему исследованию всего, что можно получить от использования event sourcing. Возможно даже больше, благодаря простой концепции хранения событий.
