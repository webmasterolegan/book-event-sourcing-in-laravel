# Корни агрегатов (Aggregate Roots)

Обычный вариант использования в любом приложении — принимать решения на основе состояния. Например: "корзину нельзя оформить, если она пуста" или "в корзине должно быть не менее 20\$ для бесплатной доставки". Мы должны обсудить, где мы будем обрабатывать такого рода бизнес-правила.

До сих пор мы взаимодействовали с агрегатами и их связанными проекциями через действия, поэтому они, вероятно, первая идея, которая приходит на ум, где обрабатывать бизнес-логику. Однако это не идеальное место: действия уже занимаются преобразованием пользовательского ввода в структурированные данные, с которыми могут работать наши события. Кроме того, такой подход разнесет основную бизнес-функциональность по нескольким классам и часто приведет к дублированию кода.

Давайте рассмотрим идею иметь одну точку входа в наш агрегат, которая знает обо всех бизнес-правилах и может взаимодействовать с нижележащим кодом. Сообщество DDD называет эту концепцию корнем агрегата (aggregate root). Вот Мартин Фаулер снова, объясняя их:

У агрегата будет один из его компонентных объектов, который будет корнем агрегата. Любые ссылки извне агрегата должны идти только к корню агрегата. Таким образом, корень может обеспечить целостность агрегата в целом.

Мы все еще можем использовать действия как способ взаимодействия с корнем агрегата, но корень агрегата является основным компонентом для взаимодействия.

Давайте посмотрим на пример: "корзину нельзя оформить, если в ней нет товаров". Изначально действие CheckoutCart выглядело бы примерно так:

```php
class CheckoutCart
{
    public function __invoke(Cart $cart): Cart
    {
        (new CartCheckedOut(
            cartUuid: $cart->uuid,
            date: now(),
        ))->handle();

        return $cart->refresh();
    }
}
```

Давайте начнем с создания нашего первого корня агрегата и переместим некоторый код. Это CartAggregateRoot:

```php
namespace App\Cart;

use Spatie\EventSourcing\AggregateRoots\AggregateRoot;

class CartAggregateRoot extends AggregateRoot
{
    public function checkout(): self
    {
        $this->recordThat(new CartCheckedOut(
            cartUuid: $this->uuid(),
            date: now(),
        ));

        return $this;
    }
}
```

Первое, на что следует обратить внимание, это то, что CartAggregateRoot наследуется от абстрактного класса AggregateRoot. Подобно проекторам и реакторам, этот класс добавляет базовую функциональность для работы с событиями. В частности, он добавляет метод `recordThat()`, который мы должны использовать вместо прямого вызова `handle()` для события.

Помимо хранения и отправки событий, `recordThat()` делает еще одну вещь: он автоматически добавляет UUID к событиям, которые он записывает. Этот UUID имеет очень важную цель, поэтому давайте обсудим его подробно.

Каждая корзина имеет свой уникальный идентификатор. Мы видели ранее, что генерируем UUID при вызове действия InitializeCart и передаем этот UUID нашему событию CartInitialized. В конечном итоге проектор использует этот UUID и создает с ним новую проекцию Cart.

Означает ли этот UUID только идентичность проекции Cart или он представляет нечто большее? Когда мы думаем о значении "корзины", мы могли бы сказать, что это больше, чем только ее проекция. "Корзина" — это результат потока событий, где проекция Cart — это просто один из побочных эффектов этого потока событий. Наша корзина — это не просто одна строка в базе данных с UUID — это набор событий и проекций. Именно этот набор мы называем "агрегатом", поэтому естественно, что UUID для "корзины" идентифицирует не только одну проекцию, но и весь агрегат в целом. Именно корень агрегата хранит этот UUID и автоматически прикрепляет его ко всем записанным событиям.

Итак, откуда берется этот UUID? Точно из того же места, что и раньше: он генерируется в действии при инициализации корзины. Вместо того чтобы напрямую передавать его событиям, мы используем его для получения корня агрегата:

```php
class InitializeCart
{
    public function __invoke(Customer $customer): Cart
    {
        $cartUuid = uuid();
        $cartAggregateRoot = CartAggregateRoot::retrieve($cartUuid);

        // return Cart::find($cartUuid);
    }
}
```

Если UUID корня агрегата прикреплен ко всем его событиям, это означает, что мы также можем получить ранее сохраненные события из базы данных для этого корня агрегата. Эти события могут быть воспроизведены внутренне каждый раз, когда извлекается корень агрегата. Звучит знакомо? Это именно то, что мы делали с событийными запросами.

Однако есть одно важное отличие: событийные запросы используются для запроса существующих событий; они представляют модель потока событий только для чтения. Однако корни агрегатов будут использовать эти предыдущие события для принятия новых решений, что означает, что состояние корня агрегата не должно быть изменено извне.

Давайте еще раз посмотрим на наш метод `checkout`: мы не можем продолжать, если корзина пуста. Вместо того чтобы запрашивать проекции для определения соблюдения наших бизнес-правил, мы можем посмотреть на сами события. Каждый раз, когда товар добавляется, мы будем отслеживать его в нашем корне агрегата. Каждый раз, когда он удаляется, мы будем делать то же самое.

Отслеживание таких изменений для построения своего внутреннего состояния выполняется в отдельных защищенных методах корня агрегата. Вам не нужно обрабатывать каждое событие, если это не необходимо, и у вас есть гибкость, чтобы вносить изменения во внутреннее состояние корня агрегата в любое время: ничего не сохраняется, поэтому вам не нужно беспокоиться о миграциях данных. Давайте представим еще два метода в нашем CartAggregateRoot: `addItem` и `removeItem`:

```php
class CartAggregateRoot extends AggregateRoot
{
    // ...

    public function addItem(
        string $cartItemUuid,
        Product $product,
        int $amount
    ): self {
        $this->recordThat(new CartItemAdded(
            cartItemUuid: $cartItemUuid,
            productUuid: $product->uuid,
            amount: $amount,
        ));

        return $this;
    }

    public function removeItem(CartItem $cartItem): self
    {
        // Проверка, существует ли товар в этой корзине...

        $this->recordThat(new CartItemRemoved(
            cartItemUuid: $cartItem->uuid,
        ));

        return $this;
    }
}
```

Эти два метода записывают события, но не применяют их к какому-либо состоянию. Помните, что корень агрегата перестраивается с нуля при каждом его извлечении, поэтому мы должны разделить "запись событий" и "управление состоянием". При воспроизведении событий должны вызываться только методы, связанные с состоянием, а не исходная запись событий.

Эти методы-слушатели выглядят именно так, как вы и ожидали, так же, как и в событийных запросах:

```php
class CartAggregateRoot extends AggregateRoot
{
    // ...

    private array $cartItems = [];

    protected function onItemAdded(CartItemAdded $event): void
    {
        $this->cartItems[$event->cartItemUuid] = true;
    }

    protected function onItemRemoved(CartItemRemoved $event): void
    {
        unset($this->cartItems[$event->cartItemUuid]);
    }
}
```

В нашем случае нам нужно только знать, пуста корзина или нет. Отслеживание добавленных товаров с помощью простого массива подходит для этой задачи.

Давайте вернемся к нашему методу `checkout()`. Теперь мы можем проверить, пуста ли корзина:

```php
class CartAggregateRoot extends AggregateRoot
{
    // ...

    public function checkout(): self
    {
        if (count($this->cartItems) == 0) {
            throw new EmptyCartException();
        }

        $this->recordThat(new CartCheckedOut(
            date: now(),
        ));

        return $this;
    }
}
```

И давайте не забудем проверить, действительно ли товар корзины существует перед его удалением (мы пропустили эту часть сначала, потому что у нас еще не было локального состояния):

```php
class CartAggregateRoot extends AggregateRoot
{
    // ...

    public function removeItem(CartItem $cartItem): self
    {
        if (! isset($this->cartItems[$cartItem->uuid])) {
            throw new CartItemNotFound();
        }

        $this->recordThat(new CartItemRemoved(
            cartItemUuid: $cartItem->uuid,
        ));

        return $this;
    }
}
```

Обратите внимание, как мы генерируем исключения из наших корней агрегатов, чтобы указать, что что-то не так. Задача корня агрегата — не определять, как обрабатывается недопустимая операция. Он должен только сигнализировать, что что-то не так. Разработчик должен обрабатывать эти исключения на другом уровне.

В некоторых случаях исключения лишь предотвращают действительно недопустимую операцию, которая могла бы произойти только в том случае, если разработчик допустил ошибку при написании кода. Такие исключения можно жестко завершать, поскольку они должны быть правильно обработаны до развертывания. Вы можете думать об этих исключениях как о последней линии защиты от самих себя, когда нам нужно вернуться к кодовой базе через несколько месяцев или даже лет после того, как мы изначально в ней работали. Вполне вероятно, что мы забудем некоторые тонкие детали нашей системы со временем, и хорошо знать, что есть страховочная сеть.

Однако другие исключения имеют обоснованную обратную связь для пользователя. Например, корзина, которую нельзя оформить, потому что она пуста, должна быть сообщена пользователю. Контроллеры могут перехватывать эти исключения и добавлять, например, ошибки валидации или flash-сообщения, чтобы донести обратную связь до пользователя. Самое главное, что корень агрегата обнаружил проблему и не должен был беспокоиться о том, как ее обработать.

По сути, мы можем думать о нашем корне агрегата как о конечном автомате: он отслеживает состояние внутренне и может принимать решения на основе этого состояния, например, генерируется ли исключение или записывается ли событие. Всякий раз, когда мы извлекаем корень агрегата, его состояние восстанавливается на основе предыдущих событий. Это также называется реконституцией (reconstitution), которую можно рассматривать как "перестроение".

Я упоминал, что UUID корня агрегата автоматически добавляется к его записанным событиям. Давайте вернемся к нашему CartProjector и посмотрим, как мы можем получить к нему доступ:

```php
class CartProjector extends Projector
{
    // ...

    public function onCartCheckedOut(CartCheckedOut $event): void
    {
        $cart = Cart::find($event->aggregateRootUuid());
    }
}
```

Каждое событие, записанное в корне агрегата, будет иметь этот метод `aggregateRootUuid()`. Хотя это удобное сокращение, иногда может быть трудно узнать, где именно было записано событие и с каким корнем агрегата оно связано.

Лично мне проще явно хранить UUID корня агрегата в событии и давать ему правильное имя, например так:

```php
class CartCheckedOut extends ShouldBeStored
{
    public function __construct(
        public string $cartUuid,
        public Carbon $date,
    ) {}
}
```

Вы можете заполнить UUID корзины из корня агрегата при записи события, используя метод `uuid()`.

```php
class CartAggregateRoot extends AggregateRoot
{
    // ...

    public function checkout(): self
    {
        if (count($this->cartItems) == 0) {
            throw new EmptyCartException();
        }

        $this->recordThat(new CartCheckedOut(
            cartUuid: $this->uuid(),
            date: now(),
        ));

        return $this;
    }
}
```

Используйте тот стиль, который лучше всего подходит вам и вашей команде.

Есть еще кое-что, что нужно упомянуть: как использовать эти корни агрегатов извне. На самом деле это довольно просто. Вот пример использования его в действии CheckoutCart:

```php
class CheckoutCart
{
    public function __invoke(Cart $cart): Cart
    {
        $cartAggregateRoot = CartAggregateRoot::retrieve($cart->uuid);
        $cartAggregateRoot->checkout()->persist();

        return $cart->refresh();
    }
}
```

Обратите внимание на метод `persist()` у корня агрегата: корень агрегата будет откладывать сохранение событий и их отправку наружу до вызова `persist()`. Сохранение гарантирует, что вы сохраняете события только после того, как все бизнес-правила были проверены. Возьмем, к примеру, действие для добавления нескольких товаров сразу:

```php
class AddCartItems
{
    public function __invoke(Cart $cart, Product ...$products): Cart
    {
        $cartAggregateRoot = CartAggregateRoot::retrieve($cart->uuid);

        foreach ($products as $product) {
            $cartAggregateRoot->addItem(
                cartItemUuid: uuid(),
                product: $product,
                amount: 1,
            );
        }

        $cartAggregateRoot->persist();

        return $cart->refresh();
    }
}
```

Если бы мы добавляли десять товаров и возникла проблема при добавлении третьего, мы не хотели бы, чтобы некоторые события уже были сохранены, а некоторые нет. Это безопасность, которую мы получаем от использования `persist()`.

## ТЕСТИРОВАНИЕ

Тестирование корней агрегатов можно проводить на нескольких уровнях. Поскольку они содержат основную функциональность вашего приложения, крайне важно правильно их тестировать.

Во-первых, существует модульное тестирование: тестирование корней агрегатов изолированно. Пакет event sourcing имеет встроенную поддержку для таких тестов, позволяя вам создать подделку (fake) корня агрегата. Такие подделки добавляют полезные тестовые методы для утверждений записанных событий, а также не запускают никакие побочные эффекты, такие как реакторы или проекторы.

Идея таких тестов следующая:

- настроить подделку корня агрегата с набором ранее записанных событий;
- сделать что-то с корнем агрегата;
- утвердить, что либо были сгенерированы исключения, либо были записаны новые события.

Вот один пример, но не забудьте также изучить набор тестов в пакете shop:

```php
/** @test */
public function can_add_item(): void
{
    CartAggregateRoot::fake(self::CART_UUID)
        ->given([
            $this->cartInitializedFactory->create(),
        ])
        ->when(function (CartAggregateRoot $cartAggregateRoot) {
            $cartAggregateRoot->addItem(
                self::CART_ITEM_UUID,
                $this->product,
                1
            );
        })
        ->assertRecorded([
            $this->cartItemAddedFactory->create(),
        ]);
}
```

Подделки корней агрегатов — это мощный инструмент для написания тестов, поэтому обязательно прочитайте документацию пакета event sourcing, когда соберетесь их использовать.

Далее, есть интеграционные тесты, которые проверяют, работает ли поток в целом. В таких тестах я люблю начинать с действий и утверждать, что проекции и другие ожидаемые побочные эффекты обработаны правильно.

```php
/** @test */
public function test_add_item(): void
{
    $product = ProductFactory::new()->create();

    $cartItem = (new AddCartItemAction)(
        cart: $this->cart,
        product: $product,
        amount: 1,
    );

    $this->assertEquals(1, $this->cart->refresh()->items->count());
    $this->assertTrue($cartItem->product->is($product));
}
```

Помните, что вам не нужны избыточные тесты, но ваша цель — покрыть как можно больше кода. Вам нужно будет балансировать между количеством модульных и интеграционных тестов на протяжении всего проекта. К сожалению, не существует фиксированной формулы, чтобы определить, нужно ли вам писать модульный тест, интеграционный или комбинацию того и другого. Вам придется принимать такие решения в каждом конкретном случае.

Однако я считаю, что стремление к максимальному покрытию с минимально возможным перекрытием между модульными и интеграционными тестами часто является отличным способом начать.
